# Feature Brief: OCR-First 强制识别与文本框校正融合

## 1. Overview
当前流程中，MinerU 与 OCR 的职责边界不够清晰，导致文本框上下边界不稳定、行拆分不一致，以及样式推导来源混杂。该功能将转换流程升级为 OCR-first：强制使用 OCR 作为文本识别主源，文本颜色/大小/行数优先从 OCR+图像采样推导；MinerU 仅用于文本框聚合参考。针对 OCR 框上下边界不准问题，引入“先构造 pad 搜索窗口、后分阶段裁剪/外扩”的 bbox 校正机制，确保文本完整包围并减少截断与误扩张。

## 2. Goals
- 将文本识别主流程切换为强制 OCR，移除 OCR 失败回退到 MinerU-only 的路径。
- 建立稳定的 OCR 文本框校正流程，显著降低文本下边缘截断问题（尤其 case3 目标句）。
- 统一样式推导来源：颜色、字号、行数均由 OCR+图像采样得到，避免混源导致的不一致。
- 将 MinerU 降级为“框聚合参考”，最终框以 OCR 框合并后的最小外接框为准。
- 通过严格样本指标实现可量化验收（case1/case3/英文拆分场景）。

## 3. User Stories

- **US-001: 强制 OCR 主流程**
  - **As a** 转换工具使用者, **I want to** 系统始终以 OCR 作为文本识别主源, **so that I can** 获得一致且可控的文本识别行为。
  - **Acceptance Criteria:**
    - [ ] OCR 初始化或推理失败时，流程直接报错并停止，不进入 MinerU-only 回退路径。
    - [ ] CLI/GUI 执行路径均遵循强制 OCR 策略。

- **US-002: OCR 文本框上下限校正**
  - **As a** 结果质量关注者, **I want to** OCR 框经过 pad+裁剪+外扩校正后再用于渲染, **so that I can** 避免文字被截断或框过大。
  - **Acceptance Criteria:**
    - [ ] 初始阶段先构造左右/上下 10% 的 pad 搜索窗口，用于一次性横向字体统计与后续边界判断。
    - [ ] 第一波仅对原始 OCR 框进行上下内向裁剪（不使用 pad 扩展区作为裁剪基准），用于消除原框过大带来的冗余边界。
    - [ ] 第二波仅对第一波未裁剪到边界的一侧进行外向 extend，使用预先构造的 pad 搜索窗口补齐原框过小问题。
    - [ ] 校正后 bbox 可稳定覆盖文本上下边界，不出现明显截断。

- **US-003: OCR 样式优先推导**
  - **As a** 使用者, **I want to** 文本颜色/字号/行数优先由 OCR 与图像采样决定, **so that I can** 获得与 OCR 结果一致的渲染。
  - **Acceptance Criteria:**
    - [ ] 颜色来源于校正后 OCR 框的图像采样。
    - [ ] 字号由校正后 OCR 框（或行框）推导。
    - [ ] 行数优先使用 OCR 结果，不再依赖旧的颜色投影主导行切分。

- **US-004: MinerU 仅作框聚合参考**
  - **As a** 算法维护者, **I want to** MinerU 仅参与重叠框聚合而不覆盖 OCR 定位, **so that I can** 保持 OCR-first 的最终定位一致性。
  - **Acceptance Criteria:**
    - [ ] OCR 与 MinerU 重叠时，不采用 MinerU 原框。
    - [ ] 最终框采用“重叠 OCR 框 + 参考 MinerU 框聚合后 OCR 框集合”的最小外接框。
    - [ ] 文本内容与样式仍以 OCR 推导结果为准。

## 4. Functional Requirements
- **FR-1**：系统必须以 OCR 为文本识别唯一主路径；OCR 失败必须终止任务并返回错误。
- **FR-2**：系统必须先构造 OCR 框的统一 pad 搜索窗口（左/右/上/下各 10%），用于一次性完成横向字体统计与后续边界搜索。
- **FR-3**：系统必须执行两阶段纵向边界校正：
  1) 第一波仅在原始 OCR 框内执行上/下裁剪，去除无字体色行；
  2) 第二波仅对第一波未裁剪到边界的一侧执行外向 extend，直到遇到无字体色行。
- **FR-4**：系统必须将颜色、字号、行数的推导来源切换为 OCR+图像采样优先。
- **FR-5**：系统必须将 MinerU 文本框用途限制为“重叠聚合参考”，不得直接覆盖 OCR 框。
- **FR-6**：系统必须在重叠场景输出最终最小外接框（基于 OCR 聚合结果）。
- **FR-7**：系统应输出调试信息与可视化产物，支持对 OCR 原框与校正框进行对比。
- **FR-8**：系统必须在测试中覆盖 case1/case3 的关键文本、框对齐、行合并与非截断验证。

## 5. Non-Goals (Out of Scope)
- 不在本阶段引入新的 OCR 模型或多模型投票机制。
- 不在本阶段实现跨段落语义重排或阅读顺序重建。
- 不在本阶段做多语言特化调参策略（按当前语言配置执行）。
- 不在本阶段保留 MinerU 样式推导逻辑作为主通路。

## 6. Success Metrics
- case1 目标标题（“基于AI的声明式数据开发新范式”）OCR校正框与 JSON 目标框 IoU ≥ 0.90。
- case1 英文目标行（合并后）与 JSON 行框 IoU ≥ 0.90，且不发生错误拆行渲染。
- case3 目标句（“AI辅助、效率大幅提升且质量高度稳定”）渲染后下边缘无可见截断。
- OCR 失败路径返回明确错误并中止（无 MinerU-only 回退）。
- 新增/更新的单元与集成测试全部通过。

## 7. Open Questions
1. “字体颜色行”判定阈值（颜色距离、最小像素占比）采用固定值还是可配置？
2. pad 10% 在超小框场景是否需要最小/最大像素钳制（避免过小或过大波动）？
3. 英文被拆分为多 OCR 框时，行内合并规则使用“同 y 带 + x 间距阈值”还是“颜色连续性优先”？
4. 最终最小外接框在重叠聚合时，是否需要设置最大扩张上限以防异常框污染？
